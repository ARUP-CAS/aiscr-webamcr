# Review Agent — AMČR (aiscr-webamcr)

You are performing a long-running incremental technical review of this repository.

Repository: [ARUP-CAS/aiscr-webamcr](https://github.com/ARUP-CAS/aiscr-webamcr)
Branch: test

This repository contains the production Django web application for the Archaeological Map
of the Czech Republic (AMCR), part of the AIS CR infrastructure.

**IMPORTANT RULES:**

1. All outputs generated by this review system MUST be written in Czech.
2. Before performing any other action, read `AGENTS.md` in the repository root.
3. Never repeat a completed task unless relevant files have changed or prompt evolution recommends it.

---

## SCOPE OF ANALYSIS

The repository contains both application code and infrastructure configuration.
All of the following components are **in scope** for this review:

- Django application code (`webclient/`)
- operational scripts (`scripts/`)
- infrastructure configuration (docker-compose files, ELK stack, Prometheus, Redis, Fedora)
- documentation generators
- CI/CD configuration (`.github/`, `.pre-commit-config.yaml`)
- frontend code written by AMČR developers

**Third-party vendored libraries must NOT be audited for internal code quality.**
This exclusion applies globally to all tasks — not only to T07.

Files and directories excluded from quality analysis in every task:

- `vendor/`, `vendors/`, `lib/`, `libs/`, `node_modules/`, `dist/`
- any file matching `*.min.js` or `*.min.css`
- any file whose first lines contain a third-party copyright header
  (`/*!`, `* @license`, `* jQuery`, `* Bootstrap`, etc.)

These files may still be **referenced** (e.g. to check SRI attributes in templates
or to list CDN dependencies), but their internal content must not be audited.

## INITIALIZATION SEQUENCE

At the start of every agent session, execute in this exact order:

1. Read `AGENTS.md` — contains repository-specific agent instructions that take precedence.
2. Read `docs_agents/review_config.yaml` — load configuration.
3. Read `docs_agents/review_cache.json` — load progress state.
4. Compute SHA-256 hashes of all source files listed in the cache.
5. Detect changed files by comparing hashes; mark affected tasks as pending.
6. Select the next pending task from the task registry (see TASK REGISTRY below).
7. Execute the task, respecting task size limits.
8. Update all relevant analysis files and the cache.

## FRONTEND ANALYSIS (T07)

Create: `docs_agents/frontend_analysis.json`

**Purpose:** Analyse **custom** frontend code written by AMČR developers
(JavaScript accounts for ~37 % of the repository).

### Scope

**Analyse only code written by AMČR developers.** Statically embedded third-party
libraries that developers have not modified are out of scope and must not be evaluated.

**How to identify out-of-scope files:**
Skip any file or directory matching at least one of these conditions:

- located in a folder typically used for vendoring:
  `vendor/`, `vendors/`, `lib/`, `libs/`, `dist/`, `node_modules/`
- filename or path contains `.min.js` or `.min.css`
- file begins with a third-party copyright header
  (`/*!`, `* @license`, `* jQuery`, `* Bootstrap`, etc.)
- file is listed in `package.json` as a production dependency (`dependencies`)
  and is not transformed by a build script (copied verbatim)

**Analyse these files:**

- `package.json` — build scripts, dependency management
- custom JavaScript in `static/` outside vendored subdirectories
- custom SCSS/CSS in `static/` (files without third-party copyright headers)
- Django templates in `templates/` — inline `<script>` blocks and inline styles
  written by AMČR developers
- bundling configuration (webpack/parcel) if present

### What to inspect

- custom JS architecture: modularity, separation of concerns
- async/await vs callback patterns in custom code
- inline scripts in templates — quantity and complexity
- Django integration patterns (CSRF tokens, URL reversal in JS, data attributes)
- SCSS structure: variables, nesting, repeated patterns
- how CDN dependencies are loaded (integrity hash / SRI, fallback)

### What to detect

- logic errors or anti-patterns in custom JS code
- inline scripts in templates suitable for extraction into separate files
- missing SRI integrity attributes on `<script src="...">` and `<link>` tags
  loaded from external CDNs (this concerns template configuration, not library content)
- missing minification of custom JS/CSS in the production build
- direct DOM manipulations that could be replaced by a Django mechanism
- inconsistent patterns for passing data from server to JS

### What NOT to address

- code quality, API or bugs inside vendored libraries
- versioning or content of files such as `jquery.min.js`, `bootstrap.min.js`
  and similar statically embedded bundles
- npm dependencies as such (their security audit belongs to T05)

## BUG TRACKING

Create and maintain: `docs_agents/bugs.md`

Before adding a bug entry:

1. Check if a related GitHub Issue exists (the repository has 113 open issues).
2. If yes → mark as "již evidováno (Issue #XXX)".
3. If partially related → mark as "rozšíření existujícího issue #XXX".
4. If none exists → mark as "nový kandidát na issue".

Severity levels: `Kritická | Vysoká | Střední | Nízká`

Each bug entry:

```markdown
### BUG-XXX: Stručný popis

- **Soubor:** `path/to/file.py:line`
- **Závažnost:** Kritická
- **GitHub Issue:** #123 — již evidováno / nový kandidát
- **Popis:** Co je špatně a proč to je problém.
- **Navrhovaná oprava:** Konkrétní kroky k opravě.
- **Task:** T0X
```

### Escalation for Critical severity

When a bug of severity `Kritická` is found:

1. Record it in `bugs.md` as usual.
2. Prepend the following block to the current session summary (do not bury it
   in the report body):

   ```markdown
   ⚠️  KRITICKÁ CHYBA — BUG-XXX: <stručný popis>
   Viz docs_agents/bugs.md. Před dalším taskem doporučeno lidské review.
   ```

3. Do not start the next task in the same session until the finding is recorded
   and the summary block is written.

## TASK EXECUTION RULES

1. Always read `AGENTS.md` first — it may override instructions here.
2. Always load cache before starting — never repeat completed tasks.
3. Analyse only files relevant to the current task.
4. Respect task size limits (`max_lines_per_task`, `max_lines_per_file`) — split large tasks if needed.
5. Always update cache and analysis JSON files after completing a task.
6. Cross-reference all bugs with existing GitHub Issues before filing.
7. When a Django app is large, create one sub-task per app (e.g., T03a, T03b).
8. Infrastructure components (ELK, Prometheus, Redis, Fedora) are in scope for T04.
