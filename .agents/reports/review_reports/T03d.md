## 1. Summary of findings — T03d

Analyzed 6 smaller applications (~789 lines total, 10 models). A total of 10 model entries were added to `orm_analysis.json`. Found 1 new bug (BUG-015: typo in `__init_`), 1 new N+1 candidate (NP1-020), and several missing indexes.

### Key findings

- **BUG-015** (Medium): `NalezPredmet.__init_` defined with a single underscore (line 116) — Python does not call this method as a constructor, so the attributes `close_active_transaction_when_finished` and `active_transaction` are never set.
- **NP1-020** (Low): `Lokalita.set_snapshots()` — calls `katastry.all()` and then `katastry.count()` (two queries instead of one).

## 2. Detailed model analysis

### komponenta/models.py

**Positive:**
- `KomponentaVazby` — a pattern analogous to `HistorieVazby` (typ_vazby + navazany_objekt property)
- `Komponenta.__init__()` correctly sets `active_transaction`, `close_active_transaction_when_finished`, `suppress_signal`
- `KomponentaAktivita` — a clean M2M table with unique_together

**Problems:**
- `Komponenta.pocet_nalezu` (property): calls `self.objekty.all().count() + self.predmety.all().count()` — 2 COUNT queries instead of one (annotation with `Count`)
- `Komponenta.get_absolute_url()`: follows 3–4 FK hops without select_related (`komponenta_vazby.dokumentacni_jednotka.archeologicky_zaznam.typ_zaznamu`)
- `Komponenta`: missing `db_index` on the `obdobi` and `areal` FKs

### lokalita/models.py

**Positive:**
- OneToOne to `ArcheologickyZaznam` as the primary_key — an efficient schema
- Custom PID management (IGSN) with `DigitalObjectIdentifierClient`
- `get_by_ident_cely()` filters via the FK (`archeologicky_zaznam__ident_cely`) — correct

**Problems:**
- `set_snapshots()` (l. 83-90): calls `katastry.all()` and then `katastry.count()` — two queries on the same queryset (see NP1-020)
- Missing `db_index` on the `druh` and `typ_lokality` FKs

### nalez/models.py

**Positive:**
- Good indexing: `komponenta`, `druh`, `specifikace`, `pocet` have `db_index=True`
- Consistent ordering by `druh__razeni`, `specifikace__razeni`

**Problems:**
- **BUG-015**: `NalezPredmet.__init_` (line 116) — the method is defined with a single underscore instead of `__init__` with two. Python does not recognize this method as a constructor, so the custom initialization (`close_active_transaction_when_finished = False`, `active_transaction = None`) is never performed. The sibling model `NalezObjekt` (l. 48) correctly has `__init__`.

### dj/models.py

**Positive:**
- All FK fields have `db_index=True` — excellent indexing
- Uses `FieldTracker` from `model_utils` for change tracking
- `__init__()` stores `initial_pian_id` — consistent with the repo pattern
- `initial_pian` property has a try/except `DoesNotExist` — safe lazy evaluation

**Problems:**
- No ORM problems found — the model is well written

### neidentakce/models.py

**Positive:**
- OneToOne to `DokumentCast` as the primary_key — an efficient schema
- `NeidentAkceVedouci` — a clean M2M table with unique_together

**Problems:**
- `__init__()` (l. 44-51): accesses `self.dokument_cast.dokument` — a potential extra query on instantiation (but protected by try/except ValueError)
- Missing `db_index` on the `katastr` FK

### notifikace_projekty/models.py

**Positive:**
- `GenericForeignKey` with a compound index on `(content_type, object_id)` — correct
- `UniqueConstraint` on `(user, content_type, object_id)` — prevents duplicates

**Problems:**
- Missing `db_index` on the `user` FK — queries for "my watchdogs" (hlídací psi) may be slow
