# T06 — Celery task analysis

**Date:** 2026-03-11
**Affected files:** `webclient/webclient/celery.py`, `webclient/webclient/settings/base.py`, `webclient/cron/tasks.py`, `webclient/notifikace_projekty/tasks.py`, `.agents/analysis/celery_analysis.json`

## 1. Summary of findings

- Celery is configured in a standard way: the broker and cache/back-end run on Redis, results are stored via `django-db`, and scheduling of periodic tasks is handled by `django_celery_beat` in the database.
- The main domain tasks (data deletion/anonymization, accessibility changes, data import, restoring Redis snapshots) are implemented as separate `@shared_task` functions; most of them have reasonable logging and, in critical cases, transactional handling.
- Minor shortcomings were identified in exception handling (long batch operations without a central `try/except`, dependence on an external service without a timeout) and a potential race/polling pattern in `check_hlidaci_pes`.
- Most periodic tasks are designed idempotently (filtering on a specific state or date), which reduces the risk of side effects on repeated execution.

## 2. Identified problems

- `cron.tasks.update_all_redis_snapshots`
  - The long-running batch pass over large tables (iterator + Redis pipeline) has no central `try/except`; an error in one iteration terminates the whole task without a summary (the number of successfully processed records).
- `cron.tasks.update_single_redis_snapshot`
  - Missing handling of the situation where a record does not exist in the DB (DoesNotExist); this is an expected case that is worth logging rather than letting it bubble up as an uncaught exception.
- `cron.tasks.update_materialized_views`
  - Multiple `REFRESH MATERIALIZED VIEW` commands are run in a single SQL string without local error logging; an error in one view terminates the whole refresh without detailed information.
- `cron.tasks.call_digiarchiv_update_task`
  - The HTTP call `requests.get(settings.DIGIARCHIV_URL)` runs without an explicit `timeout` and without `try/except`; if the target service is down, the worker can hang until the OS-level timeout.
- `notifikace_projekty.tasks.check_hlidaci_pes`
  - After the task starts, it waits in a while loop (`sleep(0.5)`) for the existence of a `Projekt` in the DB, without a global time limit; this is a polling pattern that, with a misconfiguration or a transaction rollback, may run longer than desired.

## 3. Improvement proposals

- **Long batch operations**
  - Wrap the main loop in `update_all_redis_snapshots` in a `try/except` with a summary log (count of successful records, model type), optionally add a coarse progress log (e.g. the count of classes/instances at the end).
  - In `update_single_redis_snapshot`, catch `DoesNotExist` and similar exceptions and log the missing record instead of crashing the task.
  - In `update_materialized_views`, wrap `cursor.execute(query)` in a `try/except` and log the specific error; optionally split the refresh into individual commands so that the failure of one view does not stop all of them.
- **External HTTP calls**
  - In `call_digiarchiv_update_task`, use `requests.get(url, timeout=<reasonable_value>)` and wrap it in a `try/except` with logging; consider setting a retry policy via Celery (e.g. countdown + max_retries).
- **Polling / race condition**
  - In `check_hlidaci_pes`, prefer scheduling the task via `transaction.on_commit()` (the task runs only after the transaction with the project is committed) or at least add a maximum time budget / iteration count for the while loop and terminate the task with a warning log once it is exceeded.
