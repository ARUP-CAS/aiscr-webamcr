# T07 — Frontend code analysis (JS/SCSS/templates)

**Date:** 2026-03-11  
**Affected files:** `webclient/static/js/mapa_basic_functions.js`, `webclient/static/js/mapa_functions.js`, `webclient/static/js/ajax_functions.js`, `webclient/static/js/theme-toggle.js`, `webclient/static/scss/_app-global.scss`, `webclient/static/scss/_app-theme-dark.scss`, `webclient/static/scss/_app-variables.scss`, `webclient/templates/base.html`, `.agents/analysis/frontend_analysis.json`

## 1. Summary of findings

- The frontend is built on Bootstrap 5, custom SCSS modules, and Leaflet for the map part; the JavaScript is organized into several larger files for maps and forms, without a bundler (served directly from `static/js`).
- Dark mode is implemented in a modern way (`prefers-color-scheme`, a `data-theme` attribute on `<html>`, and classes on `<body>`), with a dedicated `ThemeManager` and the `_app-theme-dark.scss` SCSS layer, which overrides CSS variables.
- The inline scripts in `base.html` handle mainly datepicker initialization, auto-hiding of flash messages, authentication checks via AJAX, and the language switcher; these are candidates for extraction into separate JS files.
- The code contains no serious security or logic errors; minor inconsistencies and refactoring opportunities were found (an unused variable in geolocation, duplicated SCSS blocks, missing minification of the custom JS).

## 2. Identified problems

- **`mapa_basic_functions.js` — minor bug in geolocation**
  - The `getLocation()` function references the variable `x`, which is not defined in the file; when called, it would lead to a JS error when writing the error message (probably an old leftover of the code).
- **`ajax_functions.js` — limited error handling**
  - The helpers for setting default values of selects (`get_vychozi_hodnota_podrazeneho`, `get_vychozi_licence`) use `XMLHttpRequest` without more detailed handling of network errors; on failure they only empty the target select.
- **`_app-global.scss` — duplicated blocks for `.formset-table`**
  - Several groups of selectors repeat the same `display: none` / `display: inherit` pattern for `.small` and `div.small`, which reduces readability and complicates future changes.
- **`base.html` — larger inline scripts**
  - The inline block implements:
    - datepicker initialization via jQuery,
    - automatic hiding of flash messages after 5 s,
    - a periodic login check via fetch to `core:check_authentication` and a possible redirect from the login page,
    - the language switcher (AJAX POST to `set_language`).
  - The logic is correct, but it is less readable and harder to reuse in inline form.
- **CDN without SRI (Google Tag Manager)**
  - `base.html` loads Google Tag Manager (`gtag.js`) directly from `www.googletagmanager.com` without SRI; for this type of first-party tool it is not critical, but it is good to have this state consciously documented.
- **Build pipeline for JS**
  - The custom JavaScript is not minified or bundled; there are several map and utility files that are loaded individually, which may slightly increase the number of requests and the amount of data transferred in production.

## 3. Improvement proposals

- **Geolocation and minor bugs**
  - In `getLocation()`, add a valid target element for the error message (or log the error to the console), or remove the function if it is no longer called anywhere.
- **AJAX helpers**
  - In `ajax_functions.js`, add at least basic error logging (`console.error`) and optionally switch to the `fetch` API for a more consistent style; possibly add a user message if the call fails.
- **SCSS refactoring**
  - In `_app-global.scss`, unify the duplicated blocks for `.formset-table` into a single readable block or create a utility/mixin; this reduces noise in the stylesheet without changing behavior.
- **Extraction of inline scripts from `base.html`**
  - Create separate files (e.g. `datepicker-init.js`, `messages.js`, `auth_check.js`, `language_switcher.js`) and load them in `base.html` via `{% static %}` so the logic is better versionable and testable.
- **Build pipeline for the custom JS**
  - Consider introducing a lightweight bundler (Webpack/Vite/rollup) or at least a minification step within the existing `django-compressor` setup to merge and minify the custom JS files (map scripts, utility functions, theme toggle).
