**Date:** 2026-03-10
**Affected files:** Complete repository structure (root, webclient/, scripts/, docs/, infrastructure)

---

## 1. Summary of findings

The `aiscr-webamcr` repository is a complex production Django web application for the Archaeological Map of the Czech Republic (AMČR). It consists of **23 Django applications**, **7 infrastructure components**, and extensive documentation (~261 RST files).

**Key statistics:**
- 23 Django applications in `webclient/`
- ~190 migrations in total (the largest count: `uzivatel` with 31 migrations)
- 77 Python files in `core/` — by far the largest application
- Some applications have their own tests; several applications (`cron`, `fedora_management`, `healthcheck`, `neidentakce`, `notifikace_projekty`, `services`, `vypis`, `xml_generator`) lack tests
- 7 GitHub Actions workflows
- 5 docker-compose variants (prod, dev, proxy, test, git-deploy)
- 7 infrastructure components (Elasticsearch, Kibana, Logstash, Prometheus, Redis, Fedora, Nginx proxy)
- 18 deployment/maintenance scripts in `scripts/` (deploy, runtime, database, and test scripts)

## 2. Identified problems

### P01 — Applications without tests
Six applications have no test coverage, and two of them contain Celery tasks or notification logic:

| Application | Problem |
|----------|---------|
| `cron` | Contains Celery tasks (tasks.py) — no tests |
| `notifikace_projekty` | Contains Celery tasks + views — no tests |
| `neidentakce` | Domain entity — no tests |
| `fedora_management` | Fedora repository management — no tests |
| `services` | Shared utilities (mailer) — no tests |
| `vypis` | Data export — no tests |
| `xml_generator` | XML export generator — no tests |
| `healthcheck` | Health check endpoint — no tests |

**Severity:** Medium (testing Celery tasks is critical for reliability)

### P02 — Concentration of logic in `core`
The `core` application contains 77 Python files and 26 migrations. It is responsible for authentication, connectors, middleware, coordTransform, management commands, and shared models. This high concentration of responsibilities is architectural debt.

**Severity:** Medium (refactoring would improve clarity, but is not urgent)

### P03 — Inconsistent script naming
The file `services/notfication_settings.py` contains a typo (`notfication` instead of `notification`).

**Severity:** Low

### P04 — `.example` configuration files without documentation of the copying process
The infrastructure components use `.example` files (elasticsearch.example.yml, kibana.example.yml, logstash.example.conf, prometheus.example.yml). The procedure for activating them should be documented.

**Severity:** Low

---

## 3. Improvement proposals

1. **Add tests to `cron` and `notifikace_projekty`** — these applications contain Celery tasks whose failures are hard to detect without tests.
2. **Consider gradually splitting `core`** — e.g. extract `coordTransform`, `ident_cely`, and `repository_connector` into separate utility modules.
3. **Fix the typo** in `services/notfication_settings.py`.
4. **Document the activation of the `.example` configuration files** in the deployment documentation.
