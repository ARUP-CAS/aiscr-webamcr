# T10 — Scripts analysis (scripts)

**Date:** 2026-03-13
**Affected files:** all files in `scripts/` (including `scripts/db/*.py`, `scripts/ci_deployment/*.sh`), `.agents/analysis/scripts_analysis.json`

## 1. Summary of findings

- The `scripts/` directory contains 18 files: 7 deploy scripts, 2 runtime (entrypoint), 6 healthcheck/maintenance (including 2 Python scripts in `scripts/db/`), 1 test/CI script, and 2 configuration files (`crontab.txt`, `uwsgi_site.ini`).
- The only script with an explicit `set -e` is `entrypoint.sh`; the other shell scripts rely on the `er()` wrapper for logging and propagating return codes, but commands outside `er()` (e.g. `docker image prune`, `jq`, calls without a check) may fail without stopping.
- There is significant duplication among the deploy scripts (`prod_deploy.sh`, `dev_deploy.sh`, `test_deploy.sh`, `git_prod_deploy.sh`): the functions `ask_continue`, `echo_dec`, `er()`, `check_create_network`, `check_stack_exists`, `check_file_exist` are repeated.
- `restore_database.sh` does not validate the variables `DBNAME`, `USED_DB_BACKUP`, `DB_FLAG_ROLE` before DROP/CREATE of the database (tracked as BUG-013).
- The Python script `scripts/db/db_connection_from_docker-web.py` does not read `DB_PORT` from the secret, so the health check always uses port 5432 (tracked as BUG-014). `clone_test_db.py` does not verify the existence of the secret or the required keys before use.

## 2. Identified problems

- **Missing set -e:** Most shell scripts do not have `set -e` at the start. A command failure outside the `er()` wrapper (e.g. `mkdir -p`, `docker image prune`, a `jq` call) does not cause immediate termination of the script.
- **restore_database.sh:** No check of required environment variables; with empty values this may lead to unintended behavior or an unclear error. No `set -e`.
- **build_prod_image.sh:** Minimal script without error handling and without documentation.
- **db_connection_from_docker-web.py:** Ignores `DB_PORT` in the JSON secret → the health check may test a different port than the one actually used.
- **clone_test_db.py:** Does not verify the existence of `/run/secrets/db_conf` or the required keys in the JSON; uses f-strings for SQL (values from the secret, low risk, but parameterization would be more consistent).
- **Duplicated logic:** The shared functions in the deploy scripts should be extracted into `scripts/common.sh`.

## 3. Improvement proposals

- Add `set -e` (and optionally `set -o pipefail`) to all deploy and maintenance shell scripts; for interactive blocks consider disabling it temporarily.
- In `restore_database.sh`, check at the start that `DBNAME`, `USED_DB_BACKUP`, `DB_FLAG_ROLE` are set; on error print usage and exit with exit 1.
- Create `scripts/common.sh` with the functions `ask_continue`, `echo_dec`, `er()`, `check_create_network`, `check_stack_exists`, `check_file_exist` and load it in the deploy scripts via `source`.
- In `db_connection_from_docker-web.py`, also read `DB_PORT` from the JSON (with a default of 5432) and pass it to `psycopg2.connect(..., port=db_port)`.
- In `clone_test_db.py`, add a check at the start for the existence of the secret file and the presence of the keys; optionally use parameterized queries where psycopg2 allows it for DDL.
