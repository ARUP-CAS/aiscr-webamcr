## 1. Shrnutí zjištění T03b

### 1.1 Opravy zjištění z T03a

**NP1-012 (SamostatnyNalez.save())** — Zjištění z T03a bylo správné: extra SELECT se provádí bezpodmínečně při každém `save()` (kde `pk is not None`). Property `initial_pristupnost` v modelu existuje, ale je nastavována *po* extra SELECT v `save()`, nikoliv v `__init__()`. Vzor je neúplný (viz BUG-004).

**pas/models.py cached_property** — OPRAVA T03a: `pas/models.py` správně importuje `from functools import cached_property`. Původní zjištění o distlib se netýká tohoto souboru — distlib import je pouze v `uzivatel/models.py` (BUG-003 zůstává platný).

### 1.2 Nová zjištění

- **BUG-004** (Nízká): SamostatnyNalez.save() extra SELECT — initial vzor neúplný
- **BUG-005** (Střední): Heslar.nazev_heslare bez db_index — globálně filtrované FK
- **BUG-006** (Nízká): get_vyskovy_bod() dvojí .count() na stejném querysetu
- `xml_generator/models.py:23-54`: `check_if_task_queued` vytváří novou instanci `Celery` app při každém volání — zbytečná inicializace
- `pian/models.py:207-240`: `get_create_org()` má triple-hop N+1: `dok_jednotka.archeologicky_zaznam.akce.projekt.organizace`
- Potvrzeno: views používají `select_related` při načítání objektů pro detail view, ale nikoliv před voláním `check_pred_*` metod

## 2. Podrobná analýza modelů

### heslar/models.py

**Pozitivní:**
- `HeslarDatace`, `HeslarDokumentTypMaterialRada`, `HeslarHierarchie`, `HeslarOdkaz` — všechny mají `__init__()` s uložením initial hodnot FK polí (správný vzor)
- `CheckConstraint` pro `HeslarHierarchie.typ` — validace na DB úrovni
- Konzistentní vzor `suppress_signal` v `__init__()`

**Problémy:**
- `Heslar.nazev_heslare` FK nemá `db_index=True` — viz BUG-005
- `HeslarHierarchie.heslo_podrazene` a `heslo_nadrazene` FK nemají `db_index=True`
- `HeslarOdkaz.heslo` FK nemá `db_index=True`
- Properties `dokument_typ_material_rada`, `podrazena_hesla`, `nadrazena_hesla` dělají DB dotazy bez cachování — pokud volány v iteraci, způsobí N+1

### pian/models.py

**Pozitivní:**
- Všechna geometry pole mají `db_index=True` (GiST indexy)
- `PianSekvence` má `UniqueConstraint` a správně používá `using="urgent"`
- `set_permanent_ident_cely()` ověřuje duplicity smyčkou — správné pro edge case s existujícími ident_cely

**Problémy:**
- `pristupnost_pom` (property, ř. 97-115): iteruje `dok_jednotky` bez `select_related('archeologicky_zaznam__pristupnost')` — N+1 pro každou DJ
- `evaluate_pristupnost_change` (ř. 125-147): stejný vzor
- `get_create_org()` (ř. 220-240): triple-hop N+1 — přistupuje k `dok_jednotka.archeologicky_zaznam.akce.projekt.organizace` bez select_related. Každý krok je potenciálně extra DB dotaz.
- `Pian.__init__()` nenastavuje `initial_stav` — chybí pro konzistenci (ale `pristupnost` se zjišťuje dynamicky, nikoliv z pole, takže extra SELECT v save() nenastane)
- `get_ZM_from_point()` (ř. 388-405): volá `len()` místo `.count()` na výsledku querysetu — ale vzhledem k `list()` wrapperu je to méně kritické

### pas/models.py

**Pozitivní:**
- `from functools import cached_property` — správný import (oprava T03a)
- `cached_property` použito pro `large_thumbnail` a `small_thumbnail`
- `CheckConstraint` pro geometrii (`samostatny_nalez_geom_check`)
- `check_pred_archivaci`, `check_pred_potvrzenim`, `check_pred_odeslanim` — jednoduché kontroly bez N+1

**Problémy:**
- `SamostatnyNalez.save()` (ř. 182-186): extra SELECT při každém uložení (viz BUG-004)
- FK pole bez `db_index=True`: `projekt`, `katastr`, `okolnosti`, `pristupnost`, `obdobi`, `druh_nalezu`, `specifikace`, `nalezce`, `predano_organizace` — 9 FK bez indexu
- `UzivatelSpoluprace`: `spolupracovnik` a `vedouci` FK bez `db_index=True`
- `nahled_soubor` property (ř. 326-335): volá `self.soubory.soubory.count()` bez cachování — ale property je voláno `cached_property` metodami, takže minimální dopad

### adb/models.py

**Pozitivní:**
- `Adb.__init__()` správně ukládá `initial_dokumentacni_jednotka`
- `FieldTracker` z `model_utils` — sledování změn polí (alternativa k initial_ vzoru)
- Dobré indexování u Adb: `typ_sondy`, `autor_popisu`, `rok_popisu`, `autor_revize`, `rok_revize`, `dokumentacni_jednotka` mají `db_index=True`

**Problémy:**
- `Adb.podnet` FK nemá `db_index=True`
- `Adb.sm5` FK nemá `db_index=True`
- `VyskovyBod.typ` FK nemá `db_index=True`
- `VyskovyBod.adb` FK nemá `db_index=True` (relace přes cascade delete)
- `get_vyskovy_bod()` (ř. 163-177): volá `vyskove_body.count()` dvakrát — viz BUG-006

### oznameni/models.py

**Pozitivní:**
- Jednoduchý model bez ORM problémů
- OneToOne na Projekt jako primary key — efektivní schéma

### xml_generator/models.py

**Pozitivní:**
- `ModelWithMetadata.__init__()` nastavuje správně `suppress_signal`, `active_transaction`, `close_active_transaction_when_finished`
- Fedora transakce jsou správně spravovány přes `on_commit()` — atomicky
- Konzistentní vzor `record_deletion()` a `save_metadata()` napříč všemi modely

**Problémy:**
- `check_if_task_queued()` (ř. 13-54): vytváří novou instanci `Celery("webclient")` při každém volání — Celery app initialization je drahá operace. Měla by být singleton nebo modul-level instance.
- `record_ident_change` + `process_arch_z` (ř. 422-443): iteruje přes `dokumentacni_jednotky_akce`, `casti_dokumentu`, `externi_odkazy` bez prefetch — ale je volána `on_commit`, takže je to oddělené od HTTP request-response cyklu a riziko pro uživatele je nižší.
- `record_deletion()` (ř. 325-345): dva oddělené for-cykly přes `soubory.soubory.all()` bez prefetch

## 3. ORM vzory ve views

### projekt/views.py — detail view (ř. 167-210)
Dobré pokrytí `select_related`:
```python
Projekt.objects.select_related("kulturni_pamatka", "typ_projektu", "vedouci_projektu", "organizace")
Akce.objects.filter(projekt=projekt).select_related("archeologicky_zaznam__pristupnost", "hlavni_typ")
```

### arch_z/views.py (ř. 839, 922)
Před `az.check_pred_odeslanim()` a `az.check_pred_archivaci()` není `az` prefetchován s dokumentačními jednotkami.

### Celkové hodnocení views
- 53 výskytů `select_related/prefetch_related` napříč 7 views.py souborů
- Detail views jsou obecně dobře optimalizovány
- Kritická mezera: check_pred_* volání nemají prefetch ve views — konfirmuje N+1 z T03a

## 6. Návrhy zlepšení (nad rámec T03a)

### 6.1 Přidat db_index na Heslar.nazev_heslare
```python
nazev_heslare = models.ForeignKey(
    "HeslarNazev", models.RESTRICT, db_column="nazev_heslare",
    db_index=True,  # přidat
    ...
)
```

### 6.2 Opravit initial_pristupnost v SamostatnyNalez.__init__()
Model nemá `__init__()` — přidat:
```python
def __init__(self, *args, **kwargs):
    super().__init__(*args, **kwargs)
    self._initial_pristupnost = self.pristupnost if self.pk else None
```
A v `save()` nahradit extra SELECT porovnáním:
```python
def save(self, *args, **kwargs):
    if self._initial_pristupnost is not None and self._initial_pristupnost != self.pristupnost:
        self.initial_pristupnost = self._initial_pristupnost
    super().save(*args, **kwargs)
```

### 6.3 Opravit dvojí count() v get_vyskovy_bod()
```python
count = vyskove_body.count()
if count == 0:
    return f"{adb.ident_cely}-V000{offset}"
elif count <= MAXIMAL_VYSKOVY_BOD + offset:
    ...
```

### 6.4 Singleton Celery app v xml_generator
```python
# Modul-level
_celery_app = None

def _get_celery_app():
    global _celery_app
    if _celery_app is None:
        _celery_app = Celery("webclient")
        _celery_app.config_from_object("django.conf:settings", namespace="CELERY")
    return _celery_app
```
