## 1. Summary of findings — T03b

### 1.1 Corrections of findings from T03a

**NP1-012 (SamostatnyNalez.save())** — The finding from T03a was correct: the extra SELECT is performed unconditionally on every `save()` (where `pk is not None`). The `initial_pristupnost` property exists in the model, but it is set *after* the extra SELECT in `save()`, not in `__init__()`. The pattern is incomplete (see BUG-004).

**pas/models.py cached_property** — CORRECTION of T03a: `pas/models.py` correctly imports `from functools import cached_property`. The original finding about distlib does not apply to this file — the distlib import is only in `uzivatel/models.py` (BUG-003 remains valid).

### 1.2 New findings

- **BUG-004** (Low): SamostatnyNalez.save() extra SELECT — incomplete initial pattern
- **BUG-005** (Medium): Heslar.nazev_heslare without db_index — a globally filtered FK
- **BUG-006** (Low): get_vyskovy_bod() double .count() on the same queryset
- `xml_generator/models.py:23-54`: `check_if_task_queued` creates a new `Celery` app instance on every call — unnecessary initialization
- `pian/models.py:207-240`: `get_create_org()` has a triple-hop N+1: `dok_jednotka.archeologicky_zaznam.akce.projekt.organizace`
- Confirmed: the views use `select_related` when loading objects for the detail view, but not before calling the `check_pred_*` methods

## 2. Detailed model analysis

### heslar/models.py

**Positive:**
- `HeslarDatace`, `HeslarDokumentTypMaterialRada`, `HeslarHierarchie`, `HeslarOdkaz` — all have `__init__()` storing the initial values of the FK fields (correct pattern)
- `CheckConstraint` for `HeslarHierarchie.typ` — validation at the DB level
- Consistent `suppress_signal` pattern in `__init__()`

**Problems:**
- `Heslar.nazev_heslare` FK has no `db_index=True` — see BUG-005
- `HeslarHierarchie.heslo_podrazene` and `heslo_nadrazene` FKs have no `db_index=True`
- `HeslarOdkaz.heslo` FK has no `db_index=True`
- The properties `dokument_typ_material_rada`, `podrazena_hesla`, `nadrazena_hesla` make DB queries without caching — if called in a loop, they cause N+1

### pian/models.py

**Positive:**
- All geometry fields have `db_index=True` (GiST indexes)
- `PianSekvence` has a `UniqueConstraint` and correctly uses `using="urgent"`
- `set_permanent_ident_cely()` verifies duplicates with a loop — correct for the edge case with existing ident_cely

**Problems:**
- `pristupnost_pom` (property, l. 97-115): iterates `dok_jednotky` without `select_related('archeologicky_zaznam__pristupnost')` — N+1 for each DJ
- `evaluate_pristupnost_change` (l. 125-147): same pattern
- `get_create_org()` (l. 220-240): triple-hop N+1 — accesses `dok_jednotka.archeologicky_zaznam.akce.projekt.organizace` without select_related. Each step is potentially an extra DB query.
- `Pian.__init__()` does not set `initial_stav` — missing for consistency (but `pristupnost` is determined dynamically, not from a field, so no extra SELECT occurs in save())
- `get_ZM_from_point()` (l. 388-405): calls `len()` instead of `.count()` on the queryset result — but given the `list()` wrapper this is less critical

### pas/models.py

**Positive:**
- `from functools import cached_property` — correct import (T03a correction)
- `cached_property` used for `large_thumbnail` and `small_thumbnail`
- `CheckConstraint` for geometry (`samostatny_nalez_geom_check`)
- `check_pred_archivaci`, `check_pred_potvrzenim`, `check_pred_odeslanim` — simple checks without N+1

**Problems:**
- `SamostatnyNalez.save()` (l. 182-186): extra SELECT on every save (see BUG-004)
- FK fields without `db_index=True`: `projekt`, `katastr`, `okolnosti`, `pristupnost`, `obdobi`, `druh_nalezu`, `specifikace`, `nalezce`, `predano_organizace` — 9 FKs without an index
- `UzivatelSpoluprace`: `spolupracovnik` and `vedouci` FKs without `db_index=True`
- `nahled_soubor` property (l. 326-335): calls `self.soubory.soubory.count()` without caching — but the property is called by `cached_property` methods, so the impact is minimal

### adb/models.py

**Positive:**
- `Adb.__init__()` correctly stores `initial_dokumentacni_jednotka`
- `FieldTracker` from `model_utils` — field-change tracking (an alternative to the initial_ pattern)
- Good indexing on Adb: `typ_sondy`, `autor_popisu`, `rok_popisu`, `autor_revize`, `rok_revize`, `dokumentacni_jednotka` have `db_index=True`

**Problems:**
- `Adb.podnet` FK has no `db_index=True`
- `Adb.sm5` FK has no `db_index=True`
- `VyskovyBod.typ` FK has no `db_index=True`
- `VyskovyBod.adb` FK has no `db_index=True` (relation via cascade delete)
- `get_vyskovy_bod()` (l. 163-177): calls `vyskove_body.count()` twice — see BUG-006

### oznameni/models.py

**Positive:**
- Simple model without ORM problems
- OneToOne to Projekt as the primary key — an efficient schema

### xml_generator/models.py

**Positive:**
- `ModelWithMetadata.__init__()` correctly sets `suppress_signal`, `active_transaction`, `close_active_transaction_when_finished`
- Fedora transactions are correctly managed via `on_commit()` — atomically
- Consistent `record_deletion()` and `save_metadata()` pattern across all models

**Problems:**
- `check_if_task_queued()` (l. 13-54): creates a new `Celery("webclient")` instance on every call — Celery app initialization is an expensive operation. It should be a singleton or a module-level instance.
- `record_ident_change` + `process_arch_z` (l. 422-443): iterates over `dokumentacni_jednotky_akce`, `casti_dokumentu`, `externi_odkazy` without prefetch — but it is called `on_commit`, so it is separated from the HTTP request-response cycle and the risk for the user is lower.
- `record_deletion()` (l. 325-345): two separate for-loops over `soubory.soubory.all()` without prefetch

## 3. ORM patterns in the views

### projekt/views.py — detail view (l. 167-210)
Good `select_related` coverage:
```python
Projekt.objects.select_related("kulturni_pamatka", "typ_projektu", "vedouci_projektu", "organizace")
Akce.objects.filter(projekt=projekt).select_related("archeologicky_zaznam__pristupnost", "hlavni_typ")
projekt.samostatne_nalezy.select_related("obdobi", "druh_nalezu", "specifikace", "nalezce", "katastr")
```
**Problem (l. 1027):** Before calling `projekt.check_pred_uzavrenim()`, the project is not prefetched with `akce_set__archeologicky_zaznam__dokumentacni_jednotky_akce`. The project is loaded with a plain `get_object_or_404(Projekt, ident_cely=ident_cely)`.

### arch_z/views.py (l. 839, 922)
Before `az.check_pred_odeslanim()` and `az.check_pred_archivaci()`, `az` is not prefetched with the documentation units.

### Overall assessment of the views
- 53 occurrences of `select_related/prefetch_related` across 7 views.py files
- Detail views are generally well optimized
- Critical gap: the check_pred_* calls have no prefetch in the views — confirms the N+1 from T03a

## 6. Improvement proposals (beyond T03a)

### 6.1 Add db_index to Heslar.nazev_heslare
```python
nazev_heslare = models.ForeignKey(
    "HeslarNazev", models.RESTRICT, db_column="nazev_heslare",
    db_index=True,  # add
    ...
)
```

### 6.2 Fix initial_pristupnost in SamostatnyNalez.__init__()
The model has no `__init__()` — add:
```python
def __init__(self, *args, **kwargs):
    super().__init__(*args, **kwargs)
    self._initial_pristupnost = self.pristupnost if self.pk else None
```
And in `save()` replace the extra SELECT with a comparison:
```python
def save(self, *args, **kwargs):
    if self._initial_pristupnost is not None and self._initial_pristupnost != self.pristupnost:
        self.initial_pristupnost = self._initial_pristupnost
    super().save(*args, **kwargs)
```

### 6.3 Fix the double count() in get_vyskovy_bod()
```python
count = vyskove_body.count()
if count == 0:
    return f"{adb.ident_cely}-V000{offset}"
elif count <= MAXIMAL_VYSKOVY_BOD + offset:
    ...
```

### 6.4 Singleton Celery app in xml_generator
```python
# Module-level
_celery_app = None

def _get_celery_app():
    global _celery_app
    if _celery_app is None:
        _celery_app = Celery("webclient")
        _celery_app.config_from_object("django.conf:settings", namespace="CELERY")
    return _celery_app
```
