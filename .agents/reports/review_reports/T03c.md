## 2. Detailed model analysis

### dokument/models.py

**Positive:**
- Well-designed compound indexes (5 variants on stav+ident_cely+...) ‚Äî excellent for filtering
- `CheckConstraint` on `DokumentCast` prevents simultaneous linking to both an archaeological record and a project
- `DokumentCast` has 6 compound indexes covering the main query patterns
- `DokumentExtraData` has a `CheckConstraint` for duveryhodnost (0‚Äì100)
- `__init__()` stores `initial_let` ‚Äî consistent with the repo pattern

**Problems:**
- `set_permanent_ident_cely()` (l. 441): `eval(i)` ‚Äî see SEC-ORM-004, BUG-001
- `check_pred_odeslanim()` (l. 352-355): for 3D documents it accesses `self.casti.all()[0].komponenty.komponenty.all()[0].obdobi` and `.areal` without prefetch ‚Äî cascading N+1 (see NP1-015)
- `set_snapshots()` (l. 569-580): iterates `dokumentautor_set` and `dokumentosoba_set` with nested `.autor.vypis_cely` / `.osoba.vypis_cely` ‚Äî N+1 for each author/person (see NP1-016)
- `thumbnail_image_file` (property, l. 534-543): loads ALL files and then sorts in Python ‚Äî inefficient; `.order_by('nazev').first()` would suffice
- `DokumentCast.initial_archeologicky_zaznam` and `initial_projekt` (properties, l. 781-794): do `objects.get()` on every access ‚Äî extra SELECT (see NP1-017, NP1-018)
- `Let`: the FK fields `datum`, `pozorovatel`, `organizace` have no `db_index=True`

### xml_generator/models.py (extended analysis)

Partially analyzed in T03b (NP1-014, OPT-001). T03c adds model-level analysis of the abstract models.

- `BaseAmcrModel`: abstract base model, imported by 10+ applications. Provides `__str__()` and the `get_ident_cely_link` property.
- `ModelWithMetadata`: abstract model with key architectural methods (`save_metadata`, `record_deletion`, `record_ident_change`). `record_ident_change` contains 7 isinstance branches for cascading metadata saving ‚Äî architecturally critical code.

### historie/models.py

**Positive:**
- 6 compound indexes on `Historie` ‚Äî they cover all the main query patterns (typ_zmeny+uzivatel+vazba, etc.)
- Covering index (`include=["id"]`) on `uzivatel` ‚Äî a modern PostgreSQL optimization
- 42 change types in CHOICES covering the entire domain

**Problems:**
- No ORM problems found ‚Äî the model is well indexed and simple

**Architectural impact:**
- `HistorieVazby` is an architecturally critical model ‚Äî used as a OneToOne FK from Projekt, Dokument, ArcheologickyZaznam, SamostatnyNalez, Pian, ExterniZdroj, User. The `navazany_objekt` property uses a `hasattr()` chain (8 branches) to find the linked object.

### ez/models.py

**Positive:**
- `ExterniZdrojSekvence` ‚Äî singleton pattern with `CheckConstraint(id=1)` and `using="urgent"`
- Consistent pattern of set_*() methods for workflow state transitions
- `check_set_permanent_ident()` ‚Äî reuse of the existing pattern for the permanent identifier

**Problems:**
- `get_perm_ez_ident()` (l. 301): `eval(i)` ‚Äî see SEC-ORM-005, BUG-001
- `ExterniZdroj`: missing `db_index` on `stav` and `typ` ‚Äî filtering by state may be slow
- `set_snapshots()` (l. 226-237): iteration over authors/editors with nested FK access ‚Äî N+1 (see NP1-019)
- `set_potvrzeny()` (l. 172-179): iterates `externi_odkazy_zdroje.all()` without prefetch ‚Äî potential N+1
