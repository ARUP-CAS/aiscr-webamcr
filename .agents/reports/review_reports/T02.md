**Date:** 2026-03-10
**Affected files:**
- `webclient/requirements.txt`

## 1. Summary of findings

The analysis revealed a dependency graph of 24 Django applications, 65 Python dependencies (requirements.txt), and 15 Docker services.

**Key findings:**

1. **Two circular dependencies** at the Python import level — projekt ↔ oznameni and adb ↔ xml_generator.
2. **The overgrown `core` application** imported by 16+ applications with 77 Python files — a violation of the single-responsibility principle.
3. **The `xml_generator` application** plays the role of a base-model library (BaseAmcrModel, ModelWithMetadata), even though it primarily serves as an XML generator — an unhealthy duality of responsibilities.
4. **requirements.txt mixes production and development dependencies** (selenium, debug-toolbar, pre-commit, coverage).
5. **No package.json** was found — frontend dependencies are probably handled via CDN (see T07).
6. **logstash** has no `depends_on: elasticsearch` in docker-compose.yml — a minor risk of a race condition at startup.

## 2. Identified problems

### 2.1 Circular dependencies

#### CIRC-01: projekt ↔ oznameni (High severity)

Mutual imports at the module level:

- `webclient/oznameni/models.py` imports `Projekt` from `projekt.models`
- `webclient/projekt/forms.py` imports from `oznameni.forms`
- `webclient/projekt/views.py` imports from `oznameni.forms`
- `webclient/projekt/urls.py` imports from `oznameni.views`

Django bootstrapping handles this for now thanks to lazy loading, but it is an architecturally unstable situation. Adding another direct import can at any time cause an `ImportError` or unexpected migration behavior.

**Recommendation:** Merge `oznameni` into `projekt` (logically, notifiers belong to projects), or extract the shared logic into `core/oznameni_shared`.

#### CIRC-02: adb ↔ xml_generator (Medium severity)

- `webclient/adb/models.py`: `from xml_generator.models import BaseAmcrModel, ModelWithMetadata`
- `webclient/xml_generator/generator.py`: `from adb.models import VyskovyBod` (module level)
- `webclient/xml_generator/generator.py`: `from adb.models import Adb` (lazy import inside a function)

The module-level import of `VyskovyBod` in generator.py is riskier — it binds xml_generator specifically to adb, even though xml_generator serves as generic infrastructure.

**Recommendation:** Move `BaseAmcrModel` and `ModelWithMetadata` into `core` or a new `amcr_base` module. Eliminate the adb import in xml_generator.

### 2.4 Docker: missing dependency logstash → elasticsearch

The `logstash` service in `docker-compose.yml` has no `depends_on: elasticsearch`, even though it sends data to Elasticsearch. With parallel container startup, Logstash may fail.

## 3. Improvement proposals

1. **Split requirements.txt** into `requirements.txt` (production), `requirements-dev.txt` (development), and `requirements-test.txt` (testing). This improves both productivity and the security of the production image.

2. **Eliminate CIRC-01** by merging `oznameni` into `projekt` — notifiers are a natural part of the project notification process. Alternatively, use Django signals for loose coupling.

3. **Split xml_generator** into:
   - `amcr_base` — BaseAmcrModel, ModelWithMetadata (moved into core or a new module)
   - `xml_generator` — purely a generator of XML outputs without back-imports to domain models

4. **Decompose core** — extract `core.permissions`, `core.logging`, `core.middleware`, `core.signals` into submodules with clear boundaries. This reduces the global impact of changes in core.

5. **Add `depends_on: [elasticsearch]` to logstash** in docker-compose.yml.

6. **Add a version for `sphinxcontrib-mermaid`** in requirements.txt.
