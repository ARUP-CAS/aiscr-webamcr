## 1. Shrnutí zjištění

Analýza odhalila graf závislostí 24 Django aplikací, 65 Python závislostí (requirements.txt) a 15 Docker služeb.

**Klíčová zjištění:**

1. **Dvě cirkulární závislosti** na úrovni Python importů — projekt ↔ oznameni a adb ↔ xml_generator.
2. **Přebujelá aplikace `core`** importována 16+ aplikacemi se 77 Python soubory — porušení principu jedné zodpovědnosti.
3. **Aplikace `xml_generator`** plní roli base-model knihovny (BaseAmcrModel, ModelWithMetadata), přestože primárně slouží jako generátor XML — nezdravá dualita zodpovědností.
4. **requirements.txt míchá produkční a vývojové závislosti** (selenium, debug-toolbar, pre-commit, coverage).
5. **Žádný package.json** nebyl nalezen — frontend závislosti jsou pravděpodobně řešeny přes CDN (viz T07).
6. **logstash** nemá `depends_on: elasticsearch` v docker-compose.yml — drobné riziko race condition při startu.

## 2. Identifikované problémy

### 2.1 Cirkulární závislosti

#### CIRC-01: projekt ↔ oznameni (Vysoká závažnost)

Vzájemné importy na úrovni modulů:

- `webclient/oznameni/models.py` importuje `Projekt` z `projekt.models`
- `webclient/projekt/forms.py` importuje z `oznameni.forms`
- `webclient/projekt/views.py` importuje z `oznameni.forms`
- `webclient/projekt/urls.py` importuje z `oznameni.views`

Django bootování toto zatím zvládá díky lazy loadingu, ale jde o architektonicky nestabilní situaci. Přidání dalšího přímého importu může kdykoli způsobit `ImportError` nebo neočekávané chování migrací.

**Doporučení:** Sloučit `oznameni` do `projekt` (logicky oznamovatelé patří k projektům), nebo extrahovat sdílenou logiku do `core/oznameni_shared`.

#### CIRC-02: adb ↔ xml_generator (Střední závažnost)

- `webclient/adb/models.py`: `from xml_generator.models import BaseAmcrModel, ModelWithMetadata`
- `webclient/xml_generator/generator.py`: `from adb.models import VyskovyBod` (modulová úroveň)
- `webclient/xml_generator/generator.py`: `from adb.models import Adb` (lazy import uvnitř funkce)

Modulový import `VyskovyBod` v generator.py je rizikovější — váže xml_generator konkrétně na adb, přestože xml_generator slouží jako generická infrastruktura.

**Doporučení:** Přesunout `BaseAmcrModel` a `ModelWithMetadata` do `core` nebo nového modulu `amcr_base`. Eliminovat import adb v xml_generator.

### 2.4 Docker: chybějící závislost logstash → elasticsearch

`logstash` service v `docker-compose.yml` nemá `depends_on: elasticsearch`, přestože posílá data do Elasticsearch. Při paralelním startu kontejnerů může Logstash selhat.

## 3. Návrhy zlepšení

1. **Rozdělit requirements.txt** na `requirements.txt` (produkce), `requirements-dev.txt` (vývoj) a `requirements-test.txt` (testování). Produktivita i bezpečnost produkčního image se tím zlepší.

2. **Eliminovat CIRC-01** sloučením `oznameni` do `projekt` — oznamovatelé jsou přirozenou součástí procesu oznamování projektu. Alternativně použít Django signály pro volné propojení.

3. **Rozdělit xml_generator** na:
   - `amcr_base` — BaseAmcrModel, ModelWithMetadata (přesun do core nebo nový modul)
   - `xml_generator` — čistě generátor XML výstupů bez zpětných importů na doménové modely

4. **Dekomponovat core** — vyčlenit `core.permissions`, `core.logging`, `core.middleware`, `core.signals` do podmodulů s jasnými hranicemi. Tím se sníží globální dopad změn v core.

5. **Přidat `depends_on: [elasticsearch]` k logstash** v docker-compose.yml.

6. **Přidat verzi pro `sphinxcontrib-mermaid`** v requirements.txt.
