## 1. Shrnutí zjištění

Repozitář obsahuje **6 Dockerfiles** a **8 docker-compose souborů** pokrývajících produkci, test, git-deploy a lokální vývoj. Celkový stack má až 15–18 služeb (web, nginx, celery, redis, ELK, Prometheus/Grafana, Fedora, PostgreSQL, cAdvisor, Selenium).

**Kladná zjištění:**
- Produkční `Dockerfile` používá korektní multi-stage build (python-builder → app-builder → runtime)
- Secrets jsou spravovány přes Docker Swarm external secrets v produkci — žádné hardcoded hesla v prod compose
- Všechny base images jsou pinned na konkrétní verze (kromě `memcached:latest` v dev)
- NGINX proxy používá `nginx-unprivileged` — správná bezpečnostní praxe
- Redis entrypoint dynamicky injectuje heslo ze secrets do redis.conf

**Klíčové problémy:**
1. **3× špatný secret injection vzor** (Grafana, Elasticsearch, Logstash) — env vars obsahují cestu k souboru nebo název secretu místo jeho hodnoty
2. **Sudo práva v produkčním kontejneru** — aplikační uživatel je člen sudo skupiny
3. **ELK 9.x v prod vs 8.x v dev** — major version gap
4. **`celery_worker` v produkci loguje DEBUG** — výkonnostní a bezpečnostní riziko
5. **Fedora Dockerfile:** vícenásobné `apt-get update`, žádný PID 1 management

## 2. Identifikované problémy

### 2.1 Bezpečnostní problémy

#### SEC-D01: sudo přístup v produkčním kontejneru (Střední)

`Dockerfile:99`:
```dockerfile
usermod -aG sudo user
```

Aplikační kontejner (Django, Celery) má uživatele v sudo skupině. Při exploitaci aplikační vrstvy (RCE, např. přes eval() — viz BUG-001) by útočník mohl eskalovat na root.

**Navrhovaná oprava:** Odebrat sudo skupinu. Pro nutné privilegované operace (cron, crontab setup) nastavit konkrétní NOPASSWD pravidla v `/etc/sudoers`.

#### SEC-D02: Grafana admin heslo nastaveno nesprávně (Střední)

`docker-compose.yml:149`, `docker-compose-test.yml:169`, `git_docker-compose.yml:144`:
```yaml
environment:
  - GF_SECURITY_ADMIN_PASSWORD=/run/secrets/grafana_admin_password
```

Grafana env var obdrží literální string `/run/secrets/grafana_admin_password` jako heslo, nikoli obsah souboru. Grafana Dashboards budou dostupné s tímto řetězcem jako heslem, ne s hodnotou Docker secretu.

**Navrhovaná oprava:**
```yaml
environment:
  - GF_SECURITY_ADMIN_PASSWORD__FILE=/run/secrets/grafana_admin_password
```

#### SEC-D03: ELASTIC_PASSWORD a LOGSTASH_INTERNAL_PASSWORD nastaveny na názvy secretů (Střední)

`docker-compose.yml:180-181, 201`:
```yaml
environment:
  - LOGSTASH_INTERNAL_PASSWORD=logstash_elastic_pass
  - ELASTIC_PASSWORD=elastic_pass
```

Elasticsearch ani Logstash nepodporují `_FILE` variantu pro tyto env vars. Výsledkem je, že Elasticsearch bootstrap heslo bude nastaveno na literál `"elastic_pass"` (název secret souboru, nikoli jeho obsah).

**Navrhovaná oprava:** Použít entrypoint wrapper skript:
```bash
export ELASTIC_PASSWORD=$(cat /run/secrets/elastic_pass)
```

#### SEC-D04: chmod 777 /tmp/ v Dockerfile-DB (Nízká)

`Dockerfile-DB:19` — přehnaně permisivní práva.

**Navrhovaná oprava:** `chmod 1777 /tmp/`

#### SEC-D05: Hardcoded heslo fcrepo-postgres v dev (Nízká)

`docker-compose-dev-local-db-all-containers.yml:265`:
```yaml
POSTGRES_PASSWORD=localdbpass1
```

Dev-only, ale porušuje zásadu nepoužívat hardcoded hesla.

### 2.2 Optimalizace

#### OPT-D01: Redundantní COPY v app-builder (Nízká)

`Dockerfile:37`:
```dockerfile
FROM python-builder AS app-builder
COPY --from=python-builder /wheels /wheels  # ← redundantní, app-builder IS python-builder
```

`app-builder` dědí z `python-builder` — `/wheels` jsou již přítomny.

#### OPT-D02: Souběžné .py i .pyc v runtime image (Nízká)

`Dockerfile:44,66`:
- `python3 -m compileall -b /code` vytváří `.pyc` soubory vedle `.py` v `app-builder`
- Runtime image kopíruje celý `/code` ze `app-builder` — obojí `.py` i `.pyc`
- `PYTHONDONTWRITEBYTECODE=1` zabrání tvorbě nových `.pyc`, ale zkompilované soubory jsou zkopírovány

Výsledkem je ~2× větší kódová část image. Efekt: mírné zvýšení startup performance, ale za cenu velikosti.

#### OPT-D03: Fedora Dockerfile — vícenásobný apt-get update (Nízká)

`fedora/Dockerfile:3-5`:
```dockerfile
RUN apt-get update
RUN apt-get install -y python3-pip
RUN apt-get update && apt-get install -y python3-requests
```

Tři vrstvové příkazy bez `apt-get clean` — zbytečné cache vrstvy.

**Navrhovaná oprava:** Sloučit do jednoho RUN bloku s `apt-get clean`.

#### OPT-D04: Fedora Dockerfile — žádný PID 1 management (Nízká)

`fedora/Dockerfile:18`:
```dockerfile
CMD /bin/sh -c "python3 /opt/fedora-init.py & catalina.sh run"
```

Dva procesy bez process supervisora — `SIGTERM` bude doručen `/bin/sh`, ne Tomcatu.

**Navrhovaná oprava:** Použít `tini` jako ENTRYPOINT nebo přesunout inicializaci do entrypoint skriptu.

#### OPT-D05: cron daemon v produkčním kontejneru (Nízká)

`Dockerfile:15` — `cron` je instalován a konfigurován v produkčním kontejneru. Kontejnery by měly být single-process; periodické úlohy patří do Celery beat scheduly.

### 2.3 Nekonzistence mezi prostředími

#### DCD-10: ELK Stack — major version gap (Střední)

| Prostředí | Verze |
|-----------|-------|
| Produkce, test, git-deploy | 9.3.1 |
| Dev (oba local compose) | 8.19.0 |

Elasticsearch 9.x přináší změny v REST API, výchozím nastavení xpack.security (povinné v 8+ single-node) a formátech indexů. Vývojáři debugují na jiné major verzi než je produkce.

#### DCD-02: Celery worker v produkci loguje DEBUG (Střední)

`docker-compose.yml:81`:
```yaml
command: celery -A webclient worker -l DEBUG -n worker1@amcr
```

DEBUG level výrazně zvyšuje objem logů a může exponovat citlivé informace (SQL dotazy, request payload).

**Navrhovaná oprava:** Změnit na `-l INFO` nebo `-l WARNING`.

#### DCD-07: Selenium v produkčním docker-compose.yml (Nízká)

`docker-compose.yml` obsahuje `selenium` service. Selenium patří výhradně do testovacího prostředí.

#### DCD-09: memcached:latest ve dev (Nízká)

`docker-compose-dev-local-db-all-containers.yml:96`:
```yaml
image: memcached:latest
```

Floating tag — nedeterministický build.

#### DCD-12: git_docker-compose.yml builduje Dockerfile-DEV pro Celery (Nízká)

`git_docker-compose.yml:78,104`:
```yaml
celery_worker:
  build:
    dockerfile: Dockerfile-DEV  # ← mělo by být Dockerfile
```

Celery worker v git-deploy prostředí (semi-produkce) builduje vývojový image.

#### DCD-13: Nekonzistentní síťová konfigurace v git_docker-compose.yml (Nízká)

`git_docker-compose.yml:127-128`:
```yaml
celery_exporter:
  networks:
    - prod-net  # ← ostatní služby v git compose používají git-reverse-proxy
```

## 3. Návrhy zlepšení

1. **Opravit secret injection pro Grafana, Elasticsearch, Logstash** (viz SEC-D02, SEC-D03) — použít `__FILE` suffix nebo entrypoint skripty. Toto je funkční chyba — admin hesla neodpovídají hodnotám secretů.

2. **Odebrat sudo skupinu z produkčního uživatele** (SEC-D01) — minimalizovat attack surface.

3. **Synchronizovat ELK verze** (DCD-10) — buď upgradovat dev na 9.x nebo downgrade prod na 8.x. Doporučeno: dev = prod verze.

4. **Změnit Celery log level na INFO** v produkci (DCD-02).

5. **Opravit Fedora Dockerfile** (OPT-D03, OPT-D04) — sloučit apt příkazy, přidat tini.

6. **Odstranit Selenium ze produkčního compose** (DCD-07) — nebo přesunout do samostatného souboru.

7. **Přidat depends_on: [elasticsearch] pro logstash** v docker-compose.yml (DCD-01, dříve T02).

8. **Pinovat memcached verzi** v dev compose (DCD-09).

9. **Opravit Celery image v git_docker-compose.yml** z Dockerfile-DEV na Dockerfile (DCD-12).
