## 1. Summary of findings

The repository contains **6 Dockerfiles** and **8 docker-compose files** covering production, test, git-deploy, and local development. The overall stack has up to 15–18 services (web, nginx, celery, redis, ELK, Prometheus/Grafana, Fedora, PostgreSQL, cAdvisor, Selenium).

**Positive findings:**
- The production `Dockerfile` uses a correct multi-stage build (python-builder → app-builder → runtime)
- Secrets are managed via Docker Swarm external secrets in production — no hardcoded passwords in the prod compose
- All base images are pinned to specific versions (except `memcached:latest` in dev)
- The NGINX proxy uses `nginx-unprivileged` — a correct security practice
- The Redis entrypoint dynamically injects the password from secrets into redis.conf

**Key problems:**
1. **3× wrong secret injection pattern** (Grafana, Elasticsearch, Logstash) — env vars contain the path to a file or the name of the secret instead of its value
2. **Sudo privileges in the production container** — the application user is a member of the sudo group
3. **ELK 9.x in prod vs 8.x in dev** — a major version gap
4. **`celery_worker` logs DEBUG in production** — a performance and security risk
5. **Fedora Dockerfile:** multiple `apt-get update`, no PID 1 management

## 2. Identified problems

### 2.1 Security problems

#### SEC-D01: sudo access in the production container (Medium)

`Dockerfile:99`:
```dockerfile
usermod -aG sudo user
```

The application container (Django, Celery) has the user in the sudo group. On an exploit of the application layer (RCE, e.g. via eval() — see BUG-001), an attacker could escalate to root.

**Recommended fix:** Remove the sudo group. For necessary privileged operations (cron, crontab setup), set specific NOPASSWD rules in `/etc/sudoers`.

#### SEC-D02: Grafana admin password set incorrectly (Medium)

`docker-compose.yml:149`, `docker-compose-test.yml:169`, `git_docker-compose.yml:144`:
```yaml
environment:
  - GF_SECURITY_ADMIN_PASSWORD=/run/secrets/grafana_admin_password
```

The Grafana env var receives the literal string `/run/secrets/grafana_admin_password` as the password, not the file contents. The Grafana dashboards will be accessible with that string as the password, not with the value of the Docker secret.

**Recommended fix:**
```yaml
environment:
  - GF_SECURITY_ADMIN_PASSWORD__FILE=/run/secrets/grafana_admin_password
```

#### SEC-D03: ELASTIC_PASSWORD and LOGSTASH_INTERNAL_PASSWORD set to secret names (Medium)

`docker-compose.yml:180-181, 201`:
```yaml
environment:
  - LOGSTASH_INTERNAL_PASSWORD=logstash_elastic_pass
  - ELASTIC_PASSWORD=elastic_pass
```

Neither Elasticsearch nor Logstash supports the `_FILE` variant for these env vars. As a result, the Elasticsearch bootstrap password will be set to the literal `"elastic_pass"` (the name of the secret file, not its contents).

**Recommended fix:** Use an entrypoint wrapper script:
```bash
export ELASTIC_PASSWORD=$(cat /run/secrets/elastic_pass)
```

#### SEC-D04: chmod 777 /tmp/ in Dockerfile-DB (Low)

`Dockerfile-DB:19` — overly permissive permissions.

**Recommended fix:** `chmod 1777 /tmp/`

#### SEC-D05: Hardcoded fcrepo-postgres password in dev (Low)

`docker-compose-dev-local-db-all-containers.yml:265`:
```yaml
POSTGRES_PASSWORD=localdbpass1
```

Dev-only, but it violates the principle of not using hardcoded passwords.

### 2.2 Optimizations

#### OPT-D01: Redundant COPY in app-builder (Low)

`Dockerfile:37`:
```dockerfile
FROM python-builder AS app-builder
COPY --from=python-builder /wheels /wheels  # ← redundant, app-builder IS python-builder
```

`app-builder` inherits from `python-builder` — `/wheels` are already present.

#### OPT-D02: Both .py and .pyc in the runtime image (Low)

`Dockerfile:44,66`:
- `python3 -m compileall -b /code` creates `.pyc` files next to `.py` in `app-builder`
- The runtime image copies the entire `/code` from `app-builder` — both `.py` and `.pyc`
- `PYTHONDONTWRITEBYTECODE=1` prevents the creation of new `.pyc`, but the compiled files are copied

The result is a ~2× larger code portion of the image. Effect: a slight increase in startup performance, but at the cost of size.

#### OPT-D03: Fedora Dockerfile — multiple apt-get update (Low)

`fedora/Dockerfile:3-5`:
```dockerfile
RUN apt-get update
RUN apt-get install -y python3-pip
RUN apt-get update && apt-get install -y python3-requests
```

Three layered commands without `apt-get clean` — unnecessary cache layers.

**Recommended fix:** Merge into a single RUN block with `apt-get clean`.

#### OPT-D04: Fedora Dockerfile — no PID 1 management (Low)

`fedora/Dockerfile:18`:
```dockerfile
CMD /bin/sh -c "python3 /opt/fedora-init.py & catalina.sh run"
```

Two processes without a process supervisor — `SIGTERM` will be delivered to `/bin/sh`, not to Tomcat.

**Recommended fix:** Use `tini` as ENTRYPOINT or move the initialization into an entrypoint script.

#### OPT-D05: cron daemon in the production container (Low)

`Dockerfile:15` — `cron` is installed and configured in the production container. Containers should be single-process; periodic tasks belong to the Celery beat schedule.

### 2.3 Inconsistencies between environments

#### DCD-10: ELK Stack — major version gap (Medium)

| Environment | Version |
|-----------|-------|
| Production, test, git-deploy | 9.3.1 |
| Dev (both local compose) | 8.19.0 |

Elasticsearch 9.x introduces changes to the REST API, the default xpack.security settings (mandatory in 8+ single-node), and index formats. Developers debug on a different major version than production.

#### DCD-02: Celery worker logs DEBUG in production (Medium)

`docker-compose.yml:81`:
```yaml
command: celery -A webclient worker -l DEBUG -n worker1@amcr
```

The DEBUG level significantly increases the volume of logs and may expose sensitive information (SQL queries, request payload).

**Recommended fix:** Change to `-l INFO` or `-l WARNING`.

#### DCD-07: Selenium in the production docker-compose.yml (Low)

`docker-compose.yml` contains a `selenium` service. Selenium belongs exclusively in the testing environment.

#### DCD-09: memcached:latest in dev (Low)

`docker-compose-dev-local-db-all-containers.yml:96`:
```yaml
image: memcached:latest
```

Floating tag — non-deterministic build.

#### DCD-12: git_docker-compose.yml builds Dockerfile-DEV for Celery (Low)

`git_docker-compose.yml:78,104`:
```yaml
celery_worker:
  build:
    dockerfile: Dockerfile-DEV  # ← should be Dockerfile
```

The Celery worker in the git-deploy environment (semi-production) builds the development image.

#### DCD-13: Inconsistent network configuration in git_docker-compose.yml (Low)

`git_docker-compose.yml:127-128`:
```yaml
celery_exporter:
  networks:
    - prod-net  # ← the other services in the git compose use git-reverse-proxy
```

## 3. Improvement proposals

1. **Fix the secret injection for Grafana, Elasticsearch, Logstash** (see SEC-D02, SEC-D03) — use the `__FILE` suffix or entrypoint scripts. This is a functional bug — the admin passwords do not match the secret values.

2. **Remove the sudo group from the production user** (SEC-D01) — minimize the attack surface.

3. **Synchronize the ELK versions** (DCD-10) — either upgrade dev to 9.x or downgrade prod to 8.x. Recommended: dev = prod version.

4. **Change the Celery log level to INFO** in production (DCD-02).

5. **Fix the Fedora Dockerfile** (OPT-D03, OPT-D04) — merge the apt commands, add tini.

6. **Remove Selenium from the production compose** (DCD-07) — or move it to a separate file.

7. **Add depends_on: [elasticsearch] for logstash** in docker-compose.yml (DCD-01, earlier T02).

8. **Pin the memcached version** in the dev compose (DCD-09).

9. **Fix the Celery image in git_docker-compose.yml** from Dockerfile-DEV to Dockerfile (DCD-12).
