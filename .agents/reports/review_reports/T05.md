## 1. Summary of findings

The security audit revealed **two high** and **four medium** categories of problems. No critical error was found. Secret management is primarily handled via Docker secrets, but there is a dangerous safety net. The largest immediate risks are:

1. **SEC-01 (High):** A wrong default fallback for DEBUG in `production.py` â€” it may cause `DEBUG=True` in production if the key is missing from the configuration file.
2. **SEC-02 (Medium):** Seemingly real Mailtrap credentials in a commit (`sample_secrets_mail_client.json`).
3. **SEC-03 (Medium):** Missing Django security headers (`SECURE_HSTS_SECONDS`, `SESSION_COOKIE_SECURE`, `CSRF_COOKIE_SECURE`).
4. **XSS-02, XSS-03, XSS-04 (Medium):** Three places in `vypis/` use `mark_safe()` with values from the database or model properties â€” they require targeted verification.

Positives:
- All raw SQL queries are parameterized (no direct SQL injection risk).
- Secrets are correctly loaded via the Docker secrets mechanism.
- CSRF middleware is active (`CsrfViewMiddleware`).
- ReCAPTCHA is deactivated only in the dev environment (intentional).
- `mark_safe()` in the forms is exclusively for hardcoded HTML widgets â€” safe.

## 2. Identified problems

### SEC-01 (High) â€” Dangerous fallback for DEBUG in production.py

**File:** `webclient/webclient/settings/production.py:3`

```python
DEBUG = get_secret("DEBUG", "True") == "True"
```

**Problem:** The default fallback value is the string `"True"`. If the `DEBUG` key is missing from the secrets file (`/run/secrets/db_conf`), the production instance starts with `DEBUG=True`. This exposes:
- full Python tracebacks to users
- the list of INSTALLED_APPS and settings in error pages
- deactivation of various Django security checks

**Recommendation:** Change the fallback to `"False"`:
```python
DEBUG = get_secret("DEBUG", "False") == "True"
```

---

### SEC-02 (Medium) â€” Mailtrap credentials in the repository

**File:** `webclient/webclient/settings/sample_secrets_mail_client.json`

```json
{
  "EMAIL_HOST_USER": "c527585b412e2a",
  "EMAIL_HOST_PASSWORD": "5dc4f8f4a8cdf0"
}
```

**Problem:** These values look like real Mailtrap sandbox credentials (not typical placeholders like `test_key` or `changeme`). Anyone with access to the repository can read captured test emails. Sandbox credentials are typically free and shared, but they have access to all captured emails of that account.

**Recommendation:** Verify whether the credentials are active. If so, invalidate/rotate them. Replace them with placeholder values like `"PLACEHOLDER_USER"` and `"PLACEHOLDER_PASSWORD"`.

---

### SEC-03 (Medium) â€” Missing Django security headers

**Files:** `webclient/webclient/settings/base.py`, `webclient/webclient/settings/production.py`

Missing settings:
- `SECURE_HSTS_SECONDS` â€” HTTP Strict Transport Security
- `SESSION_COOKIE_SECURE = True` â€” session cookie only over HTTPS
- `CSRF_COOKIE_SECURE = True` â€” CSRF cookie only over HTTPS
- `SECURE_CONTENT_TYPE_NOSNIFF = True` â€” protection against MIME type sniffing

**Note:** The NGINX proxy may provide these headers at the network layer. `proxy/default.conf` must be verified. Nevertheless, the Django Security checklist (`python manage.py check --deploy`) will report these values as failures.

**Recommendation:** Add to `production.py`:
```python
SECURE_HSTS_SECONDS = 31536000
SECURE_HSTS_INCLUDE_SUBDOMAINS = True
SESSION_COOKIE_SECURE = True
CSRF_COOKIE_SECURE = True
SECURE_CONTENT_TYPE_NOSNIFF = True
```

### XSS-02 (Medium) â€” mark_safe(value.get_ident_cely_link) in vypis/fields.py

**File:** `webclient/vypis/fields.py:363`

```python
return mark_safe(value.get_ident_cely_link)
```

**Problem:** The `get_ident_cely_link` property returns HTML with the `ident_cely` value from the database. If this property does not escape HTML characters, a stored `ident_cely` could contain an XSS payload. The `ident_cely` format is controlled by the application (alphanumeric prefix + number) â€” the direct risk is therefore low, but the pattern is architecturally dangerous.

**Recommendation:** Verify the implementation of `get_ident_cely_link` on the models where it is defined. If it returns an HTML string, use `format_html()` instead of string concatenation.

---

### XSS-03 (Medium) â€” mark_safe(new_instance) in vypis/fields.py

**File:** `webclient/vypis/fields.py:438`

```python
return mark_safe(new_instance)
```

**Problem:** `new_instance` comes from accessing attributes of a Dokument.extra_data instance. If the attribute returns user input without escaping, stored XSS is possible.

**Recommendation:** Verify what the relevant attribute returns. If it returns text, use `escape()` before `mark_safe()`, or rewrite using `format_html()`.

---

### XSS-04 (Medium) â€” mark_safe(field.get_name(instance)) in vypis/views.py

**File:** `webclient/vypis/views.py:79`

```python
sections_data[section][label] = mark_safe(field.get_name(instance))
```

**Problem:** The result of `get_name()` from a field object is wrapped in mark_safe. It depends on the implementation in each specific field class. If `get_name()` returns a field name from the instance (a DB value), XSS is possible.

**Recommendation:** Audit all implementations of `get_name()` in `vypis/fields.py`. Prefer `format_html()`.

---

### SEC-04 (Low) â€” BasicAuthentication in the REST API

**File:** `webclient/webclient/settings/base.py:577`

The REST Framework uses `BasicAuthentication` â€” the credentials are transmitted Base64-encoded with every request. Safe only over HTTPS (ensured by the proxy). TokenAuthentication is a better alternative for API clients.

---

### SEC-05 (Low) â€” AMCRAuthUser raises ValidationError instead of False

**File:** `webclient/core/authenticators.py:23-26`

```python
raise ValidationError(_, code="inactive")
```

The Django standard for `user_can_authenticate()` is to return `False` for inactive users. Raising an exception is non-standard and may cause unexpected behavior in edge cases (e.g. middleware handles the exception differently than the login view).

## 3. Improvement proposals

1. **Immediate action (SEC-01):** Change the `default_value` for DEBUG from `"True"` to `"False"` in `production.py`.
2. **Immediate action (SEC-02):** Verify and rotate the Mailtrap credentials. Replace them with a placeholder in the sample file.
3. **Short-term (SEC-03):** Add the Django security settings to `production.py`. Run `manage.py check --deploy`.
4. **Short-term (XSS-02/03/04):** Audit the `get_ident_cely_link` property and the `get_name()` implementations in `vypis/`. Rewrite the problematic places using `format_html()`.
5. **CI pipeline:** Add `pip audit` or `safety check` for continuous CVE monitoring. Add `manage.py check --deploy` as a CI step.
6. **Recommendation:** Add `SECURE_HSTS_SECONDS`, `SESSION_COOKIE_SECURE`, `CSRF_COOKIE_SECURE` to `production.py` regardless of the NGINX configuration (defense in depth).
