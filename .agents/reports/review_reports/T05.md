## 1. Shrnutí zjištění

Bezpečnostní audit odhalil **dvě závažné** a **čtyři středně závažné** kategorie problémů. Kritická chyba nebyla nalezena. Správa secrets je primárně řešena přes Docker secrets, ale existuje nebezpečná záchranná síť. Největší okamžitá rizika jsou:

1. **SEC-01 (Vysoká):** Chybný výchozí fallback pro DEBUG v `production.py` — může způsobit `DEBUG=True` v produkci pokud klíč chybí v konfiguračním souboru.
2. **SEC-02 (Střední):** Zdánlivě reálné Mailtrap credentials v commitu (`sample_secrets_mail_client.json`).
3. **SEC-03 (Střední):** Chybějící Django security hlavičky (`SECURE_HSTS_SECONDS`, `SESSION_COOKIE_SECURE`, `CSRF_COOKIE_SECURE`).
4. **XSS-02, XSS-03, XSS-04 (Střední):** Tři místa v `vypis/` používají `mark_safe()` s hodnotami z databáze nebo model properties — vyžadují cílené ověření.

Pozitiva:
- Všechny raw SQL dotazy jsou parametrizované (žádný přímý SQL injection risk).
- Secrets jsou správně načítány přes Docker secrets mechanismus.
- CSRF middleware je aktivní (`CsrfViewMiddleware`).
- ReCAPTCHA je deaktivována pouze v dev prostředí (záměrné).
- `mark_safe()` ve formulářích je výhradně pro hardcoded HTML widgety — bezpečné.

## 2. Identifikované problémy

### SEC-01 (Vysoká) — Nebezpečný fallback pro DEBUG v production.py

**Soubor:** `webclient/webclient/settings/production.py:3`

```python
DEBUG = get_secret("DEBUG", "True") == "True"
```

**Problém:** Výchozí hodnota fallbacku je string `"True"`. Pokud klíč `DEBUG` chybí v secrets souboru (`/run/secrets/db_conf`), produkční instance se spustí s `DEBUG=True`. Toto vystavuje:
- úplné Python tracebacky uživatelům
- list INSTALLED_APPS a settings v chybových stránkách
- deaktivaci různých bezpečnostních kontrol Django

**Doporučení:** Změnit fallback na `"False"`:
```python
DEBUG = get_secret("DEBUG", "False") == "True"
```

---

### SEC-02 (Střední) — Mailtrap credentials v repozitáři

**Soubor:** `webclient/webclient/settings/sample_secrets_mail_client.json`

```json
{
  "EMAIL_HOST_USER": "c527585b412e2a",
  "EMAIL_HOST_PASSWORD": "5dc4f8f4a8cdf0"
}
```

**Problém:** Tyto hodnoty vypadají jako reálné Mailtrap sandbox credentials (ne typické placeholdery jako `test_key` nebo `changeme`). Kdokoliv s přístupem k repozitáři může číst zachycené testovací e-maily. Sandbox credentials jsou typicky bezplatné a sdílené, ale mají přístup ke všem zachyceným e-mailům tohoto účtu.

**Doporučení:** Ověřit zda jsou credentials aktivní. Pokud ano, invalidovat/rotovat. Nahradit za placeholder hodnoty jako `"PLACEHOLDER_USER"` a `"PLACEHOLDER_PASSWORD"`.

---

### SEC-03 (Střední) — Chybějící Django bezpečnostní hlavičky

**Soubory:** `webclient/webclient/settings/base.py`, `webclient/webclient/settings/production.py`

Chybějící nastavení:
- `SECURE_HSTS_SECONDS` — HTTP Strict Transport Security
- `SESSION_COOKIE_SECURE = True` — session cookie pouze přes HTTPS
- `CSRF_COOKIE_SECURE = True` — CSRF cookie pouze přes HTTPS
- `SECURE_CONTENT_TYPE_NOSNIFF = True` — ochrana před MIME type sniffing

**Poznámka:** NGINX proxy může tyto hlavičky zajišťovat na síťové vrstvě. Je nutno ověřit `proxy/default.conf`. Nicméně Django Security checklist (`python manage.py check --deploy`) bude hlásit tyto hodnoty jako selhání.

**Doporučení:** Přidat do `production.py`:
```python
SECURE_HSTS_SECONDS = 31536000
SECURE_HSTS_INCLUDE_SUBDOMAINS = True
SESSION_COOKIE_SECURE = True
CSRF_COOKIE_SECURE = True
SECURE_CONTENT_TYPE_NOSNIFF = True
```

### XSS-02 (Střední) — mark_safe(value.get_ident_cely_link) ve vypis/fields.py

**Soubor:** `webclient/vypis/fields.py:363`

```python
return mark_safe(value.get_ident_cely_link)
```

**Problém:** Property `get_ident_cely_link` vrací HTML s hodnotou `ident_cely` z databáze. Pokud tato property neescapuje HTML znaky, mohl by uložený `ident_cely` obsahovat XSS payload. Formát `ident_cely` je řízen aplikací (alfanumerický prefix + číslo) — přímé riziko je proto nízké, ale pattern je architektonicky nebezpečný.

**Doporučení:** Ověřit implementaci `get_ident_cely_link` na modelech kde je definována. Pokud vrací HTML string, použít `format_html()` místo string concatenation.

---

### XSS-03 (Střední) — mark_safe(new_instance) ve vypis/fields.py

**Soubor:** `webclient/vypis/fields.py:438`

```python
return mark_safe(new_instance)
```

**Problém:** `new_instance` pochází z přístupu k atributům Dokument.extra_data instance. Pokud atribut vrací uživatelský vstup bez escapování, hrozí stored XSS.

**Doporučení:** Ověřit co vrací příslušný atribut. Pokud vrací text, použít `escape()` před `mark_safe()`, nebo přepsat pomocí `format_html()`.

---

### XSS-04 (Střední) — mark_safe(field.get_name(instance)) ve vypis/views.py

**Soubor:** `webclient/vypis/views.py:79`

```python
sections_data[section][label] = mark_safe(field.get_name(instance))
```

**Problém:** Výsledek `get_name()` z field objektu je opatřen mark_safe. Závisí na implementaci v každé konkrétní field třídě. Pokud `get_name()` vrací název pole z instance (DB hodnotu), hrozí XSS.

**Doporučení:** Auditovat všechny implementace `get_name()` v `vypis/fields.py`. Preferovat `format_html()`.

---

### SEC-04 (Nízká) — BasicAuthentication v REST API

**Soubor:** `webclient/webclient/settings/base.py:577`

REST Framework používá `BasicAuthentication` — přihlašovací údaje jsou přenášeny Base64-enkódované s každým požadavkem. Bezpečné pouze nad HTTPS (zajištěno proxy). TokenAuthentication je lepší alternativa pro API klienty.

---

### SEC-05 (Nízká) — AMCRAuthUser raises ValidationError místo False

**Soubor:** `webclient/core/authenticators.py:23-26`

```python
raise ValidationError(_, code="inactive")
```

Django standard pro `user_can_authenticate()` je vrátit `False` pro neaktivní uživatele. Vyhazování výjimky je nestandardní a může způsobit neočekávané chování v edge cases (např. middleware zachytí výjimku jinak než login view).

## 3. Návrhy zlepšení

1. **Okamžitá akce (SEC-01):** Změnit `default_value` pro DEBUG z `"True"` na `"False"` v `production.py`.
2. **Okamžitá akce (SEC-02):** Ověřit a rotovat Mailtrap credentials. Nahradit za placeholder v sample souboru.
3. **Krátkodobě (SEC-03):** Přidat Django security settings do `production.py`. Spustit `manage.py check --deploy`.
4. **Krátkodobě (XSS-02/03/04):** Auditovat `get_ident_cely_link` property a `get_name()` implementace v `vypis/`. Přepsat problematická místa na `format_html()`.
5. **CI pipeline:** Přidat `pip audit` nebo `safety check` pro kontinuální CVE monitoring. Přidat `manage.py check --deploy` jako CI krok.
6. **Doporučení:** Přidat `SECURE_HSTS_SECONDS`, `SESSION_COOKIE_SECURE`, `CSRF_COOKIE_SECURE` do `production.py` bez ohledu na NGINX konfiguraci (defense in depth).
