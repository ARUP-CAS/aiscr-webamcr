# T08 — Documentation and generators analysis

**Date:** 2026-03-11  
**Affected files:** `readthedocs.yaml`, `docs/generate_module_docs.py`, `docs/generate_selenium_test_docs.py`, `docs/licenses/convert_to_rst.py`, several RST files in `docs/source`, `.agents/analysis/documentation_analysis.json`

## 1. Summary of findings

- The documentation is built on Sphinx with a fairly rich structure (sections 01–13), including detailed module documentation for the Django applications, asynchronous tasks, monitoring/logging, and testing.
- The Read the Docs configuration correctly sets up the environment (Ubuntu 22.04, Python 3.12, GDAL libraries) and uses `webclient/requirements.txt` as the source of Python dependencies for the build.
- The documentation generators (`generate_module_docs.py`, `generate_selenium_test_docs.py`, `licenses/convert_to_rst.py`) are relatively robust (they use AST, sentinel markers in RST, content-change checks), but some of them could handle error states better.
- I found no obvious broken links or sections evidently inconsistent with the current code structure; real verification of correctness will happen mainly during the Sphinx/RTD builds.

## 2. Identified problems

- **Read the Docs uses the full `requirements.txt`**
  - `.readthedocs.yaml` installs `webclient/requirements.txt`, which also includes development/testing dependencies (Selenium, debug-toolbar, Sphinx, etc.); for a documentation build this is unnecessary and may slow down builds and increase the risk of conflicts.
- **`docs/licenses/convert_to_rst.py` — missing error handling**
  - The script calls `pip-licenses` via `subprocess.run` and, without checking `returncode`, directly parses `output.stdout` via `json.loads`; on a tool error or invalid output it ends with a traceback without a friendly error message.
- **Selenium documentation coverage depends on discipline in the docstrings**
  - `generate_selenium_test_docs.py` assumes that all Selenium tests have a docstring and the required `Steps` and `Expected` sections; a missing section leads to the loss of that test's documentation without the script explicitly flagging it in CI.

## 3. Improvement proposals

- **Separating requirements for the documentation**
  - Create a separate file (e.g. `docs/requirements.txt`) with the minimal set of dependencies needed to build the documentation (Sphinx, theme, possibly a few helpers) and switch the installation in `.readthedocs.yaml` to that file.
- **Improving the license generator**
  - In `convert_to_rst.py`, add a check of `output.returncode` and handle exceptions from `json.loads`; on error, print a clear message (e.g. "pip-licenses is not installed" or "invalid JSON") and return a non-zero exit code so the failure is visible in CI.
- **Strengthening the Selenium documentation check**
  - Add a step to `generate_selenium_test_docs.py` that, after parsing, evaluates missing required sections / docstrings and prints a summary report (optionally returning a non-zero exit code if the script is run in CI).

## 5. Prompt improvement proposals

- In the T08 assignment it would be useful to explicitly mention that generators based on `subprocess` (e.g. license scripts) should also be checked from the error-handling perspective (`returncode`, invalid output).
- Add that for test documentation (Selenium), the agent should explicitly assess whether the generator enforces the presence of key sections and how it behaves in case of a violation (e.g. a recommendation to wire it into CI).
